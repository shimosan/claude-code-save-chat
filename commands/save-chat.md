# /save-chat — 会話を Obsidian に保存 (新規作成 / 改訂)

## 設定

**保存先ベースディレクトリ**: `~/.claude/CLAUDE.md` の「ホスト情報」セクションの `vault_path` を参照する (移行前の端末で旧 `~/.claude/CLAUDE.local.md` が残っていればそちらでも可)。

実行時の西暦を使って `<vault_path>/claude{YYYY}/` サブフォルダに保存。例: 2026 年なら `<vault_path>/claude2026/`。フォルダが存在しなければ `mkdir -p` で作成。

## 引数

- 引数なし: 会話の主題から自動で slug を決定
- `/save-chat <slug>`: 指定した slug を使用 (改訂モード判定の鍵にもなる)

## 実行手順

### 共通フロー

1. **slug の決定**: 引数があればそれを採用、なければ会話の主題から自動生成 (規則は下記)
2. **既存ファイルの探索**: `claude{YYYY}/` (今年フォルダ) → `claude{YYYY-1}/` (前年フォルダ) の順に `*-{slug}.md` を Glob で検索
3. **モード分岐**:
   - ヒット → **改訂モード**
   - ヒットなし → **新規モード**

### 新規モード

1. `mkdir -p {年フォルダ}` でフォルダ確保
2. 表記揺れ防止: `Grep` で `claude{YYYY}/*.md` の `tags:` を抽出し、既存タグ語彙を把握
3. **テンプレート選択**: 会話の性質を見て、`default` で書くか別テンプレが合うか判断。default 以外を使う場合、ユーザーに「<性質> なので `<template名>` が合いそうですが採用しますか？」と提案して承認を得る
4. 選択したテンプレートに従って本体 (概要〜関連) を組み立てる
5. ヘッダ (frontmatter + H1) とフッタ (改訂履歴) を共通フォーマットで添える
6. `Write` で `{date}-{slug}.md` を書き出し
7. 改訂履歴に「`YYYY-MM-DD HH:mm` (初版): 初回作成」を記録 (default 以外なら末尾に「(テンプレート: <名前>)」を付記)
8. 保存パスを markdown link 形式で報告

### 改訂モード

1. 既存ファイルを `Read` し、frontmatter・本体・改訂履歴を把握
2. 新内容を生成し直す。本体構造は既存テンプレートを基本踏襲。テンプレート自体を変更する場合は大改訂扱いで、ユーザーに事前確認
3. **改訂規模の判定** (詳細は下記):
   - `小改訂`: タグ調整、誤字、節内の小規模追記、表現修正
   - `大改訂`: セクション追加/削除、結論・課題の内容変更、本文の 30% 超書き換え、テンプレート変更
4. **改訂履歴エントリの追加**:
   - 既存最新行が**今日**で**かつ今回も小改訂** → その行を更新 (回数 `×N` をインクリメント、時刻を最新に、要点をカンマ追記)
   - それ以外 (日跨ぎ or 大改訂) → 新規行を append
5. frontmatter の `last_revised` を更新 (他の初版系フィールドは変更しない)
6. `Write` で上書き保存 (重複ファイル `-2`, `-3` は作らない)
7. 「改訂しました: <path> ({小改訂|大改訂})」と報告

## slug の命名ルール

- **形式**: 小文字 ASCII の kebab-case (`claude-code-memory-sync`)
- **語数**: 3〜6 単語
- **品詞**: 主題を表す**名詞句**を基本 (動詞形は避ける)
- **語順**: 主題 → サブトピック
- **禁止語**: `note`, `discussion`, `meeting`, `chat` などジェネリックすぎる単語
- **既存 slug 参照**: `claude{YYYY}/` の既存ファイル名を Glob で確認し、語彙の流儀を踏襲

## タグの命名ルール

- **形式**: 小文字 ASCII の kebab-case
- **第 1 タグは固定**: `claude-chat`
- **追加タグ数**: 2〜4 個
- **既存タグ優先**: `Grep -h "tags:" claude{YYYY}/*.md` 相当で語彙を抽出し、近いものがあれば**新規造語せず採用**
- **新規タグを作る前のチェック**: 既存タグに同義語・上位概念がないか確認
- **参考カテゴリ** (絶対ではない、表記揺れ防止の目安):
  - ツール系: `claude-code`, `cursor`, `obsidian`, `git`, `github`, `dropbox`, `python`
  - アスペクト系: `memory`, `sync`, `history`, `permissions`, `settings`, `multi-device`, `team`
  - 活動系: `design`, `troubleshoot`, `workflow`, `setup`, `research`, `learning`, `automation`
  - メタ系: `ai-ethics`, `productivity`, `security`

## 改訂規模の判定基準

| 判定 | 条件 (いずれか該当) |
|---|---|
| `小改訂` | 誤字修正、表現調整、タグ/wikilink 修正、節内の小追記、結論を変えない補足 |
| `大改訂` | セクション追加/削除、結論や課題の内容変更、本文の概ね 30% 以上の書き換え、論旨の変更、**テンプレート変更** |

判定は AI が新旧本文を比較して機械的に決める。曖昧な場合は**小改訂寄り**に倒す。

## wikilink のスコープ規則

vault フォルダの分類とアクセスルールは `~/.claude/CLAUDE.md` の「Obsidian vault 管理」を参照。
具体的なフォルダ名は `~/.claude/CLAUDE.md` の「ホスト情報」セクションの `vault_path` / `ai_note_folder` / `public_note_folders` / `private_note_folders` で定義。

**自動でリンクを張る範囲**:
- `<vault_path>/<ai_note_folder>` 配下: `[[ファイル名]]` で自動 wikilink 可
- `<vault_path>/<public_note_folders>` 配下: 自動 wikilink 対象に含めてよい

**自動対象外** (ユーザー明示指示時のみ参照):
- `<vault_path>/<private_note_folders>` 配下
- 秘密情報を含む可能性があるため、AI は**勝手に Read しない**

**`private_note_folders` 参照時の制約**:
- ファイル名一覧の取得 (`Glob`) は OK。本文の `Read` はユーザー明示指示時のみ
- 本文を save-chat 出力に**転記する前**に「<note名> から <要点> を引用しますがよろしいですか?」と確認
- wikilink 対象に含める場合も同様に確認

**書いてよい wikilink の判定**:
1. Glob で存在確認できた `ai_note_folder` / `public_note_folders` 配下の note → `[[ファイル名]]` で書いて良い
2. `private_note_folders` 配下 → ユーザー明示指示時のみ
3. 存在不明 → wikilink にせず、本文中で言及するか、関連セクションから省略

**wikilink の書式 (重要)**:
- **裸の `[[ファイル名]]` で書く** — Obsidian のリンクとして解決させるため
- **バッククォート (`` ` ``) で囲わない** — 囲うとインラインコードとしてレンダリングされリンクが機能しない
  - NG: `` `[[2026-05-22-foo]]` `` ← コード扱いになる
  - OK: `[[2026-05-22-foo]]` ← リンクとして機能する
- ファイル名そのものを「コードとして引用したい」場合のみバッククォート使用 (ただし wikilink にはならないので、リンクが欲しければバッククォートなしで書く)

## ノートの構造 (3 zone)

すべての save-chat 出力は次の 3 zone で構成される:

| Zone | 内容 | テンプレート間の変動 |
|---|---|---|
| **ヘッダ (固定)** | frontmatter + H1 タイトル | テンプレに関係なく共通 |
| **本体 (可変)** | 概要〜関連の全域 (=「議論を整理した中身」) | **構造そのものが変わる** (節数・順序・サブ構造・書式すべて自由) |
| **フッタ (固定)** | 改訂履歴セクション | テンプレに関係なく共通 |

→ テンプレートとは「セクション名の置換」ではなく、**本体 zone の書き方の流派**を指す。Q&A 反復、ステップログ、ADR 風テーブル、症状-原因-対処の流れ、など構造そのものが異なる。

## ヘッダ (frontmatter + タイトル) — 共通フォーマット

```markdown
---
created: {初版作成日時 YYYY-MM-DD HH:mm}
last_revised: {最新改訂日時 YYYY-MM-DD HH:mm。初版時は created と同じ}
template: {default | qa | worklog | troubleshoot | decision | custom}
source: claude-code
model: {初版時のモデル ID。例: claude-opus-4-7}
session_id: {初版時の JSONL UUID。cwd の encoded path から ~/.claude/projects/<encoded>/ 内の最新 mtime の .jsonl から取得。不明なら省略}
workspace: {初版時の cwd (絶対パス)}
machine: {初版時の hostname (短縮形、`hostname -s` の出力)}
tags:
  - claude-chat
  - {2〜4 個の追加タグ。既存タグ語彙を優先}
---

# {会話の主題を端的に表すタイトル}
```

### frontmatter の更新規則

| フィールド | 初版時 | 改訂時 |
|---|---|---|
| `created` | 設定 | 変更しない |
| `last_revised` | created と同じ | **毎回更新** |
| `template` | 設定 | 通常変更しない (大改訂でテンプレ変更時のみ可) |
| `source` | `claude-code` | 変更しない |
| `model` | 設定 | 変更しない (初版時のモデル) |
| `session_id` | 設定 | 変更しない (初版時のセッション) |
| `workspace` | 設定 | 変更しない (初版時の cwd) |
| `machine` | 設定 | 変更しない (初版時のホスト) |
| `tags` | 設定 | 改訂で追加・削除可 (主題が変わるなら更新) |

→ `created` 系は「最初の記録時点のコンテキスト」を凍結保存する不変フィールド。改訂時の状況は「改訂履歴」で時系列に記録する。

## 本体 (可変) — テンプレート集

本体 zone はテンプレートに従って組み立てる。テンプレートは**目安**であり絶対ではない。会話の性質に応じて構造を変えてよい。新しい流派を使いたい場合、AI はユーザーに提案 → 承認後に本ファイルへ追記して語彙固定する。

### `default` (議論型) — 議論を時系列またはテーマ別に整理する。最も汎用

```markdown
## 概要

{2〜3 文で「何を議論したか」を要約}

## 議論

{論理展開を箇条書きで漏らさず追う。
- 重要な前提・観察・判断・反論はすべて含める
- 発言者は区別しない (自分用ノート)
- 時系列でもテーマ別でもよい
- 詳細寄り。1〜2 画面分。冗長は削るが論理の飛びは禁物
- 必要に応じて `### N. 小見出し` で分節}

## 結論

{再利用可能な知識・判断・推奨を箇条書き}

## 課題

{宿題・未解決論点。なければセクションごと省略可}

## 関連

{実在のパス・URL、`claude*/` 内の存在確認済み wikilink}
```

### 候補テンプレート (未確定、必要時に AI が提案 → ユーザー承認 → 本ファイル追記で確定)

以下は「構造の流派が異なる」例。実際に使う前に提案フェーズを挟む。

#### `qa` (Q&A型) — 独立した質疑が複数並ぶ会話

```markdown
## 概要

{何を調べた会話か。テーマを 1 文}

## Q1. {質問の見出し}

**A:** {回答本文}

{必要なら補足・例・参照}

## Q2. {別の質問}

**A:** ...

{Q3, Q4, ... と反復}

## 関連
```

#### `worklog` (作業ログ型) — コマンド実行と結果を時系列で残す

```markdown
## 概要

{何を達成しようとしたか。目的を 1〜2 文}

## 前提

- 環境: ...
- 初期状態: ...

## 実行ログ

### Step 1. {目的}
- 操作: `command here`
- 結果: ...
- 観察: ...

### Step 2. ...

## 最終状態

{完了時の状態を簡潔に}

## 課題

{残った宿題}

## 関連
```

#### `troubleshoot` (障害対応型) — 症状から原因・対処まで追跡

```markdown
## 概要

{1 文の問題ステートメント}

## 症状

{観測された挙動・エラー出力 (抜粋)}

## 再現手順

1. ...
2. ...

## 仮説と検証

| # | 仮説 | 検証方法 | 結果 |
|---|---|---|---|
| 1 | ... | ... | 棄却 |
| 2 | ... | ... | 採用 |

## 原因

{確定した原因}

## 対処

{取った対処と効果}

## 教訓

{同類問題への一般化}

## 関連
```

#### `decision` (設計判断型 / ADR) — 何かを決めた会話を判断記録として残す

```markdown
## 概要

{何を決めたか。1 文}

## 文脈

{なぜ決める必要があったか。背景・制約}

## 選択肢

| 案 | 概要 | 利点 | 欠点 |
|---|---|---|---|
| A | ... | ... | ... |
| B | ... | ... | ... |
| C | ... | ... | ... |

## 決定

**採用**: 案 X

{採用理由}

## 帰結

- **Positive**: ...
- **Negative**: ...

## 関連
```

→ これら以外の流派が必要になったら、AI が「<性質> なので `<新テンプレ名>` が合いそうですが採用しますか？」と提案。承認後に本ファイルへ追記して語彙を固定する。

## フッタ (改訂履歴) — 共通フォーマット

すべての save-chat 出力の末尾に置く。テンプレートに関係なく共通。

```markdown
## 改訂履歴

- {YYYY-MM-DD HH:mm} (初版): 初回作成
```

### 改訂履歴の記法

#### 初版時
```
- 2026-05-19 15:54 (初版): 初回作成
```

default 以外のテンプレートを使った場合:
```
- 2026-05-19 15:54 (初版): 初回作成 (テンプレート: qa)
```

#### 同日中の小改訂 (連続) — 最新行を更新 (新規行を作らない)

回数を `×N` で表記、時刻は最新に更新、要点はカンマ区切りで末尾に追記:
```
- 2026-05-19 15:54 (初版): 初回作成
- 2026-05-19 17:00 (小改訂×3): タグ修正、wikilink 削除、結論に 1 項目追加
```

単発 (×1) の場合は `×1` を省略:
```
- 2026-05-19 16:30 (小改訂): タグ表記揺れ修正
```

#### 日跨ぎの小改訂 — 新規行を append

```
- 2026-05-19 17:00 (小改訂×3): タグ修正、wikilink 削除、結論に 1 項目追加
- 2026-05-22 10:15 (小改訂): 課題に 1 項目追加
```

#### 大改訂 — 必ず新規行で記録

```
- 2026-08-15 14:00 (大改訂): ~/.claude 同期実装の実体験を反映、結論を書き直し、新セクション「実装メモ」を追加
```

テンプレート変更を伴う大改訂は明示:
```
- 2026-09-01 11:00 (大改訂): テンプレートを default → troubleshoot に変更、本体を症状-原因-対処の構造に再構成
```

## 補足

- このコマンドはどのワークスペースから起動しても同じ場所に保存される (user global commands)
- 保存先 `<vault_path>/claude{YYYY}/` はクラウドストレージ同期下なら別端末からも閲覧可能
- Obsidian vault のルートが `<vault_path>` (`CLAUDE.md` の「ホスト情報」セクションで定義) なので、保存後すぐ Obsidian で検索・wikilink できる
- 改訂モードで `-2`, `-3` の重複ファイルは作らない
